1. Document Overview
1.1 Purpose
This Design Requirements Document (DRD) translates the PhysioConnect Product Requirements Document into actionable design specifications. It defines the complete user experience architecture, interaction patterns, screen inventory, component library requirements, and design constraints for both product portals. Engineering, QA, and product teams should treat this document as the single source of design truth for all v1.0 deliverables.


1.2 Product Summary
PhysioConnect is a SaaS platform bridging physiotherapy clinics with their patients through two integrated portals:


Portal
	Primary Users
	Core Value Proposition
	Clinic Management Portal (CMP)
	Physios, Clinic Admins, Receptionists
	Streamline intake, treatment planning, exercise assignment, billing & outcomes tracking
	Patient-Facing App (PFA)
	Patients (web + mobile)
	Self-guided recovery with AI chatbot, video exercises, progress tracking, and easy clinic communication
	

1.3 Design Principles
These five principles govern every design decision across both portals:


* Clarity over density — Clinical environments are high-stress. Every screen must communicate its purpose within 3 seconds.
* Progressive disclosure — Surface only what is needed for the current task. Advanced options live behind deliberate reveals.
* Evidence-first UX — All AI outputs and exercise recommendations link back to clinical evidence. Trust is built through transparency.
* Mobile-first, desktop-complete — Patients primarily use mobile; clinicians primarily use desktop. Both must be exceptional.
* Accessibility by default — WCAG 2.1 AA compliance across all components. Not a post-launch concern.


1.4 Scope
In Scope (v1.0):
* Clinic Management Portal — full web application
* Patient App — responsive web + PWA (iOS & Android install prompts)
* AI Chatbot widget — embedded in Patient App, powered by RAG pipeline
* Exercise video library UI — browsing, playback, and assignment flows
* MSK Focus: Neck and Shoulder conditions for initial launch


Out of Scope (v1.0):
* Native iOS / Android apps
* Billing & insurance claims module (design placeholder only)
* Multi-clinic / franchise management
* Video telehealth / live consultation

Four primary personas drive all design decisions. Each profile includes goals, frustrations, and design implications.


2.1 Dr. Priya Sharma — Physiotherapist
Persona: Dr. Priya Sharma
Role: Senior Physiotherapist, 8 years experience
Context: Sees 12–18 patients/day in a mid-size Mumbai clinic
Devices: Desktop (90%), mobile for quick notes


Goals:
  • Document sessions quickly without interrupting the consultation
  • Assign tailored home exercise programmes in under 2 minutes
  • Track patient adherence and recovery trajectories across a caseload
  • Access clinical guidelines at point of care without leaving the patient record


Frustrations:
  • Current paper/Excel systems cause double-entry and lost data
  • Generic exercise PDFs are ignored by patients
  • No visibility into what patients do between sessions


Design Implications:
  • Keyboard-first interactions, dense information layout acceptable
  • Session notes must be fast — templates + voice input
  • Exercise assignment UI must allow bulk select + customise

Context: Manages 3 physiotherapists, 60+ active patients
Devices: Desktop (primary), tablet at front desk


Goals:

  • Generate weekly and monthly reports for the clinic owner


Frustrations:
  • Patients calling to confirm / cancel appointments
  • Manual reminders via WhatsApp are unreliable
  • Report generation takes 2–3 hours at month end


Design Implications:
  • Calendar must support drag-and-drop rescheduling

4. User Flows
This section documents the critical-path user flows that must be designed and validated before any other flows. Each flow includes entry states, decision points, and exit states.


4.1 Flow 1: Patient Onboarding (CMP)
Entry Point
Admin or Physio clicks &#x201C;Add New Patient&#x201D; from either the Dashboard or Patient List.


Flow Steps
1. Receptionist clicks + New Patient
2. Step 1 — Personal Info (name, DOB, contact, emergency contact)
3. Step 2 — Medical History (conditions, medications, allergies, previous physio)
4. Step 3 — Presenting Complaint (chief complaint, pain scale, onset, mechanism)
5. Step 4 — Consent & Documents (digital signature for treatment consent, data consent)
6. Step 5 — Confirm & Create — system creates patient record + generates intake PDF
7. Patient receives SMS/WhatsApp with portal login credentials


Decision Points
* If patient is a minor → guardian signature required on Step 4
* If patient opts out of data use → AI chatbot is disabled in their PFA account


Exit States
* Success: Patient record created, clinician assigned, patient notified
* Incomplete: Draft saved, banner shows &#x201C;Incomplete intake — resume here&#x201D;


4.2 Flow 2: Session Documentation (CMP)
8. Physio opens patient record via Schedule or Patient List
9. Clicks &#x201C;Start Session&#x201D; — session timer begins
10. Selects or confirms Assessment Template (MSK Neck, MSK Shoulder, or custom)
11. Fills Subjective (patient-reported pain, function change) using structured fields + free text
12. Fills Objective (ROM measurements, special tests, palpation findings)
13. Selects Assessment diagnosis from coded dropdown + adds clinical notes
14. Defines Plan — treatment today, exercise modifications, next session goals
15. Clicks &#x201C;Assign Exercises&#x201D; → launches Exercise Assignment Flow (Flow 3)
16. Reviews and signs off session note
17. System auto-saves; note appears in patient timeline


4.3 Flow 3: Exercise Assignment (CMP)
18. Physio clicks &#x201C;Assign Exercises&#x201D; from session note or patient record
19. Exercise Library opens as a right-panel drawer (does not navigate away)
20. Physio filters by: body region, condition, difficulty, equipment
21. Selects exercises — each card shows thumbnail, name, duration, difficulty chip
22. Clicks &#x201C;Add to Programme&#x201D; — exercises queue in right panel
23. For each selected exercise: sets / reps / hold time / rest / frequency — all editable inline
24. Optionally adds written cues or flags as &#x201C;check pain level before performing&#x201D;
25. Clicks &#x201C;Save Programme&#x201D; → exercises appear in patient&#x2019;s PFA immediately
26. Patient receives push notification: &#x201C;Your programme has been updated&#x201D;


4.4 Flow 4: Patient Daily Exercise (PFA)
27. Patient opens app — Today view shows pending exercises with completion rings
28. Taps first exercise card
29. Exercise Detail loads — video autoplays (muted) with instruction overlay
30. Patient taps &#x201C;Start Set&#x201D; — countdown timer begins
31. After each set: quick pain check (0–10 slider, optional)
32. After all sets: completion confirmation — &#x201C;Great work! 1 of 5 done.&#x201D;
33. Patient continues to next exercise or returns to Today view
34. On completion of all exercises: celebration animation + streak update
35. Progress data syncs to CMP; physio sees adherence in real time


4.5 Flow 5: AI Chatbot Interaction (PFA)
36. Patient taps chat bubble or Messages > AI tab
37. Welcome message + suggested questions appear
38. Patient types or selects question (e.g., &#x201C;My neck hurts after exercise, is that normal?&#x201D;)
39. RAG pipeline retrieves relevant clinical guideline chunks
40. AI responds with: plain-language answer, evidence source citation, safety flag if needed
41. If response includes safety flag → &#x201C;Consult your physio before continuing&#x201D; is appended
42. If patient asks a question outside chatbot scope → graceful handoff to clinic messaging
43. All chat sessions are logged and accessible by the assigned physio in CMP

7. Interaction Design & Motion
Motion and transitions reinforce the platform&#x2019;s clinical professionalism while providing clear feedback. All animations respect prefers-reduced-motion.


7.1 Motion Principles
* Purposeful — every animation communicates a state change or hierarchy
* Fast — UI transitions max 300ms; data loading states under 200ms trigger
* Consistent — identical elements animate identically across all screens
* Reduced motion fallback — all decorative animations disabled when OS prefers-reduced-motion is set

8. Accessibility Requirements
PhysioConnect targets WCAG 2.1 Level AA compliance for all screens. The following requirements are non-negotiable for launch.


8.1 Colour & Contrast
* Normal text (< 18pt): minimum 4.5:1 contrast ratio against background
* Large text (≥ 18pt bold or ≥ 24pt): minimum 3:1 contrast ratio
* UI components and graphical objects: minimum 3:1 contrast ratio
* Do not use colour as the only means of conveying information (e.g. error states must also include icon + text)
* High-contrast mode option in patient app settings


8.2 Touch & Interaction Targets
* Minimum interactive target size: 44×44px (iOS) / 48×48dp (Android) for all tappable elements
* Spacing between adjacent targets: minimum 8px
* Pain scale slider must be keyboard-navigable (arrow keys ± 1 unit)


8.3 Keyboard & Screen Reader
* All interactive elements reachable by Tab key in logical DOM order
* Visible focus indicator on all focusable elements (2px solid primary colour)
* Modal and drawer traps focus within; ESC closes and returns focus to trigger
* All images have meaningful alt text; decorative images have empty alt
* All form fields have associated labels (not just placeholder text)
* Error messages are announced to screen readers via aria-live regions
* Video player has: keyboard controls, closed captions, audio description for demonstration videos


8.4 Content Readability
* Patient-facing copy targets Flesch-Kincaid Grade 7 or lower
* AI chatbot responses: no jargon without inline definition
* Clinical terms in patient app display with a (?) tooltip explaining the term
* Date formats use DD Month YYYY (e.g. 05 June 2026) — never ambiguous numeric format

10.2 Error Message Framework
All error messages follow a 3-part structure: What happened + Why it happened + What to do.


* Form validation errors: inline, below the field, in danger colour with error icon
* API errors: toast notification + optional modal for critical failures
* Empty states: illustrated, with primary CTA to resolve the empty state
* 404 / not found: friendly page with navigation options back to safety


10.3 AI Chatbot Content Rules
* All AI responses cite the clinical source (e.g. &#x201C;Source: NICE Guidelines for Neck Pain, 2023&#x201D;)
* If confidence is low: &#x201C;I&#x2019;m not certain about this — please verify with your physiotherapist.&#x201D;
* Absolute guardrails — never respond to: medication dosing, surgical advice, diagnosis, mental health crisis. Redirect to qualified professionals.
* Pain > 7/10 response always includes: &#x201C;Stop activity. Contact your physio or seek medical attention if severe.&#x201D;
